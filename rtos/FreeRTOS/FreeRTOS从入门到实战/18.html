第18讲：资源管理与临界区
临界区概念、任务临界区API与调度器挂起

一、临界区概念

在嵌入式多任务系统中，临界区（Critical Section）是指一段必须独占式访问的代码区域。当多个任务或中断服务程序（ISR）可能同时访问共享资源（如全局变量、外设寄存器、内存缓冲区）时，如果不加以保护，就会导致数据不一致、状态混乱等严重问题。

典型问题场景

  • 数据竞争：两个任务同时读写一个全局变量，导致数据损坏。
  • 非原子操作：一个需要多个步骤完成的操作（如先读后写）被中途打断。
  • 硬件状态不一致：多个任务配置同一外设的不同寄存器，产生冲突配置。
注意：临界区保护的目标是共享资源，而不是代码本身。同一段代码在不同上下文中可能不需要保护。

二、任务临界区 API

FreeRTOS 提供了 taskENTER_CRITICAL() 和 taskEXIT_CRITICAL() 宏来定义临界区。它们通过操作处理器的中断屏蔽寄存器来实现。

工作原理

  1. taskENTER_CRITICAL()：关闭可屏蔽的中断（通常是 configMAX_SYSCALL_INTERRUPT_PRIORITY 以下优先级的中断）。
  2. 执行受保护的代码。
  3. taskEXIT_CRITICAL()：恢复之前的中断状态（不是简单开中断，而是恢复之前的状态）。
// 示例：使用临界区保护共享变量
static uint32_t g_sharedCounter = 0;

void TaskIncrement(void *pvParameters) {
    while(1) {
        // 进入临界区（关闭中断）
        taskENTER_CRITICAL();
        
        // 对共享资源进行独占访问
        g_sharedCounter++;
        // 可能的多步操作
        if(g_sharedCounter > 1000) {
            g_sharedCounter = 0;
        }
        
        // 退出临界区（恢复中断）
        taskEXIT_CRITICAL();
        
        vTaskDelay(pdMS_TO_TICKS(10));
    }
}

特性与限制

特性 说明 注意事项
嵌套支持 支持嵌套调用，内部维护嵌套计数器 确保 enter/exit 成对调用
中断响应 关闭部分或全部中断（取决于配置） 临界区应尽量短，避免影响中断响应
任务切换 不会发生任务切换（因为中断被关闭） 不能调用可能引起阻塞的API（如 vTaskDelay）

三、挂起调度器

当需要保护一段较长的代码区域，或者需要调用可能引起阻塞的FreeRTOS API时，可以使用 vTaskSuspendAll() 和 xTaskResumeAll() 来挂起调度器。

vTaskSuspendAll()

  • 挂起任务调度器，阻止上下文切换
  • 不关闭中断，中断服务程序仍可执行
  • 中断中引起的任务切换请求会被挂起
  • 支持嵌套调用，内部维护挂起计数器
  • 可以在临界区内调用

xTaskResumeAll()

  • 恢复任务调度器
  • 当挂起计数器减为0时，真正恢复调度
  • 返回 pdTRUE 表示恢复了挂起的上下文切换
  • 必须与 vTaskSuspendAll() 成对调用
// 示例：使用调度器挂起保护复杂操作
void ComplexOperation(void) {
    // 挂起调度器，防止任务切换
    vTaskSuspendAll();
    
    // 执行可能较长的操作
    ProcessMultipleData();
    
    // 可以安全调用某些FreeRTOS API
    UBaseType_t uxHighWaterMark = uxTaskGetStackHighWaterMark(NULL);
    
    // 恢复调度器
    BaseType_t xYieldPending = xTaskResumeAll();
    
    // 如果恢复过程中有挂起的切换请求，手动触发切换
    if(xYieldPending == pdTRUE) {
        taskYIELD();
    }
}

最佳实践建议：
  • 临界区应尽可能短，只包含必须独占访问的代码
  • 优先使用互斥信号量，除非性能要求极高
  • 在中断服务程序中只能使用 taskENTER_CRITICAL_FROM_ISR
  • 避免在临界区内调用任何可能引起阻塞的函数
  • 考虑使用静态分析工具检查临界区使用是否正确

五、综合示例

下面是一个综合应用示例，展示了在不同场景下如何选择合适的保护机制：

// 场景1：快速更新共享计数器（使用临界区）
void UpdateCounter(void) {
    taskENTER_CRITICAL();
    g_systemCounter++;
    if(g_systemCounter == 0) {
        g_rolloverCount++;
    }
    taskEXIT_CRITICAL();
}

// 场景2：复杂数据结构操作（使用调度器挂起）
void ProcessDataBuffer(void) {
    vTaskSuspendAll();  // 挂起调度器
    
    // 操作多个共享缓冲区
    for(int i = 0; i < BUFFER_COUNT; i++) {
        ProcessBuffer(&g_buffers[i]);
    }
    
    // 可以安全调用FreeRTOS API
    UBaseType_t uxTasks = uxTaskGetNumberOfTasks();
    
    // 恢复调度器
    if(xTaskResumeAll() == pdTRUE) {
        taskYIELD();  // 如果有挂起的切换，立即切换
    }
}

// 场景3：混合使用（临界区+调度器挂起）
void CriticalSystemUpdate(void) {
    // 第一步：快速更新硬件寄存器（需要关中断）
    taskENTER_CRITICAL();
    HW_REGISTER = CONFIG_VALUE;
    taskEXIT_CRITICAL();
    
    // 第二步：长时间的数据处理（挂起调度器）
    vTaskSuspendAll();
    PerformLongCalculation();
    UpdateMultipleStructures();
    xTaskResumeAll();
    
    // 第三步：再次快速更新
    taskENTER_CRITICAL();
    HW_REGISTER2 = FINAL_VALUE;
    taskEXIT_CRITICAL();
}

调试与验证

  • 栈溢出检查：在调度器挂起期间调用 uxTaskGetStackHighWaterMark() 检查栈使用
  • 时间测量：使用 traceTASK_SWITCHED_IN/OUT 钩子函数测量临界区执行时间
  • 断言检查：配置 configASSERT() 检查临界区嵌套深度和调度器状态
  • 运行时统计：启用 configGENERATE_RUN_TIME_STATS 分析保护机制的开销
