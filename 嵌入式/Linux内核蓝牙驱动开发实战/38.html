平台差异对比

特性 x86平台 ARM平台
架构 CISC (复杂指令集) RISC (精简指令集)
字节序 小端序 (Little Endian) 可配置 (通常小端序)
内存管理 统一内存架构 可能包含非一致性内存访问
设备发现 主要通过PCI/ACPI 主要通过设备树
电源管理 ACPI标准 设备树+平台特定实现

设备树适配

设备树是ARM平台硬件描述的核心机制，用于替代x86平台的ACPI和BIOS检测机制。

蓝牙设备树节点示例

&bluetooth {
    compatible = "broadcom,bcm43438-bt";
    max-speed = <115200>;
    shutdown-gpios = <&gpio 45 GPIO_ACTIVE_HIGH>;
    host-wakeup-gpios = <&gpio 22 GPIO_ACTIVE_HIGH>;
    device-wakeup-gpios = <&gpio 23 GPIO_ACTIVE_HIGH>;
    clocks = <&clk32k>;
    clock-names = "lpo";
    status = "okay";
};

关键属性说明

  • compatible: 驱动匹配字符串，用于绑定对应的驱动程序
  • max-speed: UART通信的最大波特率
  • shutdown-gpios: 蓝牙模块关机控制引脚
  • host-wakeup-gpios: 主机唤醒引脚
  • device-wakeup-gpios: 设备唤醒引脚
  • clocks: 蓝牙模块所需的时钟源

硬件差异处理

1. GPIO配置差异

x86平台通常通过PCI配置空间管理GPIO，而ARM平台通过设备树和pinctrl子系统。

/* ARM平台GPIO配置示例 */
static int bt_configure_gpios(struct bt_data *info)
{
    int ret;
    
    /* 配置关机GPIO */
    ret = devm_gpio_request_one(&pdev->dev, info->shutdown_gpio,
                    GPIOF_OUT_INIT_HIGH, "bt shutdown");
    if (ret)
        return ret;
        
    /* 配置唤醒GPIO */
    ret = devm_gpio_request_one(&pdev->dev, info->device_wake_gpio,
                    GPIOF_OUT_INIT_LOW, "bt device wake");
    if (ret)
        return ret;
        
    return 0;
}

2. 时钟管理差异

ARM平台需要显式管理时钟，而x86平台时钟通常由BIOS/固件管理。

static int bt_clock_configure(struct bt_data *info)
{
    /* 获取32K时钟 */
    info->clk = devm_clk_get(&pdev->dev, "lpo");
    if (IS_ERR(info->clk)) {
        dev_warn(&pdev->dev, "failed to get lpo clock\n");
    } else {
        clk_prepare_enable(info->clk);
    }
    
    return 0;
}

CONFIG_BT=y

CONFIG_BT_HCIUART=y
CONFIG_BT_HCIUART_BCM=y
CONFIG_BT_BCM=y

交叉编译环境设置

# 设置交叉编译工具链
export ARCH=arm
export CROSS_COMPILE=arm-linux-gnueabihf-
export KERNEL_SRC=/path/to/arm-kernel

# 编译命令
make -C $(KERNEL_SRC) M=$(PWD) modules

常见问题排查

  • 设备未识别: 检查设备树配置和GPIO引脚
  • 通信失败: 检查UART配置和波特率
  • 电源管理问题: 检查时钟和电源控制引脚
  • 驱动加载失败: 检查内核配置和依赖项

最佳实践

移植成功的关键因素：
  1. 充分理解硬件差异: 深入研究目标平台的硬件特性
  2. 设备树配置准确: 确保所有GPIO、时钟、中断配置正确
  3. 模块化设计: 将平台相关代码与通用代码分离
  4. 充分的测试: 在不同场景下全面测试驱动功能
  5. 文档完善: 记录移植过程中的关键决策和问题解决方案
